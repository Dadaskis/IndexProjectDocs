This is a documentation generator script that:
1. Parses GDScript files to extract class definitions, properties, methods, signals etc.
2. Organizes scripts by type (Node, Resource, Class, Other)
3. Generates HTML documentation pages for each module/script
4. Creates navigation sidebars and index pages
5. Handles special cases like autoloads and inherited types
The output is a complete web-based HTML documentation for this Godot project

Documentation Generator Algorithm:

1. INITIALIZATION:
- Set up data structures ( ScriptData , ModuleScriptsData )
- Define node/resource class names for categorization

2. SCRIPT PARSING PHASE:
a) Recursively find all .gd files in modules/addons
b) For each script:
- Extract metadata (class name, extends, description)
- Parse variables/constants (with export detection)
- Parse methods (including return types/args)
- Parse signals and enums
- Categorize as Node/Resource/Class/Other
- Cache parsed data to avoid re-processing

3. TYPE RESOLUTION PASS:
- Deep check inheritance chains to verify script types
- Handle special cases (autoloads, unnamed classes)

4. DOCUMENTATION GENERATION:
- Create module directories and copy template files
- For each module:
- - Generate index.html with module overview
- - For each script:
- - - Create HTML page using template
- - - Populate with parsed metadata
- - - Format code blocks with syntax highlighting
- - Build sidebar navigation with script hierarchy

5. SEARCH SYSTEM GENERATION:
- Generate JavaScript sidebar files
- Create categorized search entries (nodes/resources/etc)
- Include all script members (methods/variables/etc)

6. FINALIZATION:
- Update main documentation index
- Format all HTML with consistent indentation
- Verify all links and assets

Key Features:
- Works entirely offline
- Potentially can be hosted on some website
- Preserves code comments in output
- Handles Godot-specific cases (signals, exports)
- Generates interconnected HTML docs with search
- Supports modules/addons with custom structure

Note for searchbars:
- Why is there " [O] " in paths of searchbar_contents ?
It is simply a way to easily replace paths to fit into file hierarchy,
as you can see here:
searchbar_contents.replace(" [O] ", "../../")) # Correct path for a module HTML

Also, you need to know that this entire thing mostly works by replacing
the text from templates. The list of templates:
"/_guide/addon_template/docs/index.html" - HTML template for an addon page
^^^ This one is very similar to a module page template
"/_guide/module_template/docs/index.html" - HTML template for a module page
"/_guide/script_template/script_name.html" - HTML template for a script page

Why are there different templates? Firstly, different pages might have
some differences in their visual outlook, hence why they use different
HTML code. Additionally, their file storage uses different kind of
hierarchy, and given the fact all HTML pages are using relative paths,
different amount of "../" s might be needed.

FILE STRUCTURE:
"/_guide/" - Template files and assets
"/_guide/.images/" - Images
"/modules/ [module] /docs/" - Generated module documentation
"/addons/ [addon] /docs/" - Generated addon documentation

TEMPLATE REPLACEMENT TAGS (written [like_that] ):
module_name - Module/addon name
module_description - Description from info.tres
script_name - Script filename
script_class_name - class_name value
script_type_name - ScriptType as string
sidebar_content - Generated navigation HTML
[O] - Path placeholder (replaced with various amount of ../)

KNOWN LIMITATIONS:
- Doesn't document scene-embedded scripts
- No support for inner classes
- No cross-module linking in docs
- No documentation versioning

Function

prepare_searchbar_contents

None

any

Generates the complete search sidebar content for all documentation.
This one is hellish. I wanted to make HTML page that works entirely offline
and it imposes some of the problems. Firstly, you need to know that
generating hundreds of thousands of HTML code lines isn't such a good
idea for a searchbar for a documentation of that size. It takes ages to
generate, imposes a lot more of other problems and god knows what else.
Hence why i came to a conclusion that making a simple JS code trick would
do the justice in a way. Basically, instead of inserting HTML code into
each module/script/etc HTML file, i decided to use JS file to insert that
HTML code (look at /_guide/sidebar_injector_script.js for an example).
So, that being said, this method is *kinda* easy to understand. You just
need to know this is a glorified HTML code generator that puts that code
into a JavaScript file that is being called out by other HTML files.

Algorithm:
1. Iterate through all modules to grab the info from them
2. Add entries (simple HTML generation part here)
3. Use JS template (sidebar_injector_template.js) to make files specifically
for modules/scripts/etc

Why using a JS template? Firstly, sidebar_injector_template.js provides
easier way to "replace" certain part of JS code with the HTML
that i need to insert into these pages. In this case, this method is simply
reading this file and replacing " [sidebar_content] " with the searchbar's
HTML while keeping the rest of the code untouched. And the rest of the code
simply supports the logic of the searchbar, such as the text input
field that shows you search results as well as HTML insertion logic.

Why are there "sidebar_injector_module.js" and "sidebar_injector_script.js"?
Sadly, in this HTML page, it makes more sense to use relative paths, and
modules or scripts are all using different hierarchies that result in
different paths. So, the only difference between these two files is that
one is using "../../" and another is using "../../../"

Why is there " [O] "/" [OG] " in paths of searchbar_contents?
It is simply a way to easily replace paths to fit into file hierarchy,
as you can see here:
searchbar_contents.replace(" [O] ", "../../")) # Correct path for a module HTML
